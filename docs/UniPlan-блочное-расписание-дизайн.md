# UniPlan — блочное (цикловое) расписание как первоклассная фича: дизайн

> Цель: заложить **single-subject intensive block scheduling** (одна дисциплина изучается каждый день в течение блока, затем следующая) в ядро UniPlan — не как обходное решение через date patterns (как в UniTime), а как нативную модель данных и решателя.

См. также: [`UniTime-ограничения-и-критерии.md`](./UniTime-ограничения-и-критерии.md) — откуда взяты ограничения/критерии классического расписания, которые мы адаптируем под блоки.

---

## 0. Требования реального вуза (исходные)

Один вуз ведёт **два типа расписания одновременно** — оба должна поддерживать одна система:

1. **Младшие курсы — традиционное:** несколько дисциплин параллельно в течение периода (классическая недельная сетка). В нашей модели — `CourseOffering.scope = SPAN`.
2. **Старшие курсы — цикловое/блочное:** одна дисциплина «до упора» каждый день (пример: практика в клинике ежедневно 8:00–12:00), потом другая дисциплина, в другое время. В нашей модели — `scope = BLOCK`.

→ Гибрид — **жёсткое требование**, а не опция. Модель `scope: BLOCK | SPAN` покрывает оба в одной системе.

### Нет фиксированной сетки времени (важно!)
У занятий **нет стандартного звонкового расписания**: первая пара то в 8:30, то в 9:00; длительность занятий разная.

- **Движок это тянет:** CPSolver работает на **5-минутных слотах (288/сутки)** — произвольное начало и переменная длительность представимы нативно. «Пара» = N слотов с любым стартом. (Тип `Exact Time`: дни + время начала задаются прямо на занятии.)
- **Следствие для нас:** свободное время — **режим по умолчанию**, а не исключение. UX-слой UniTime построен вокруг стандартных time patterns (там Exact Time «для редких случаев») — поэтому мы **строим своё приложение** вокруг движка, а не форкаем приложение UniTime.
- **Время как вход или как решение:** занятие может иметь **фиксированное** время (клиника 8:00–12:00 — задано) либо **свободное** (solver выбирает старт в рабочем окне с шагом 5 мин). CPSolver покрывает оба: фиксированное = одно возможное размещение в домене; свободное = много размещений.

### Новые сущности, которые добавляет эта сверка
- **`Department/ClinicalBase`** — ресурс-кафедра/база с ёмкостью (сколько групп одновременно) по блокам (R10).
- **Разбиение offering дисциплины по кафедрам с диапазоном номеров групп** (R11): `DisciplineAssignment{disciplineId, departmentId, groupRange, capacity}`.
- **`DatePatternGenerator`** — из календаря праздников + правила периодичности (каждая N-я неделя, нечёт/чёт) генерит date pattern в датах (R3–R6).
- **Online-локация** (R12); **группа как единица спроса** (R13); **версионирование + MPP** (R14, пост-MVP).
- Параметры дисциплины ротации: `groupsParallel`, `durationDays` (R9).

## 1. Ключевая идея: две оси времени

Классическое расписание (UniTime) живёт в **одной** оси: «день недели × время» с повторением по неделям. Блок там — лишь пересечение `time pattern ∩ date pattern`, эмерджентный артефакт.

Блочное расписание имеет **две независимые оси**:

```
        ОСЬ 1: последовательность блоков (макро)
        ┌──────────┬──────────┬──────────┬──────────┐
        │  Блок 1  │  Блок 2  │  Блок 3  │  Блок 4  │   ← семестр = упорядоченная цепочка блоков
        │ (3 нед.) │ (3 нед.) │ (4 нед.) │ (3 нед.) │
        └──────────┴──────────┴──────────┴──────────┘
             │
             ▼  ОСЬ 2: дневная сетка внутри блока (микро)
        ┌─────┬─────┬─────┬─────┬─────┐
        │ Пн  │ Вт  │ Ср  │ Чт  │ Пт  │
   1пара│  A  │  A  │  A  │  A  │  A  │   ← внутри блока дисциплина A идёт каждый день
   2пара│  A  │  A  │  A  │  A  │  A  │
        └─────┴─────┴─────┴─────┴─────┘
```

**Следствие для архитектуры:** решатель должен быть **двухуровневым** — макро (что в каком блоке) и микро (расписание дня внутри блока). Это даёт огромный выигрыш: микро-задачи блоков **независимы и параллелятся**, а конфликтов внутри блока почти нет (студент учит одну дисциплину).

`Block` — сущность первого класса: упорядоченная, с датами, длиной, со своей дневной сеткой. Всё остальное (offering, секция, план студента, пререквизит) ссылается на блоки.

## 2. Доменная модель

Псевдо-схема (стек-агностично; читается как ER-модель или TS-типы).

### Календарь и структура
```
Term {
  id
  name                      // "2026/2027 осень"
  blocks: Block[]           // упорядоченная цепочка
}

Block {
  id
  termId
  sequence: int             // порядок: 1, 2, 3...  (определяет ось 1)
  name?                     // "Блок A", "Модуль 1"
  startDate, endDate        // непересекающиеся, идут подряд
  lengthDays: int           // блоки могут быть разной длины
  dayGrid: DayGrid          // дневная сетка (ось 2)
}

DayGrid {                   // микро-сетка внутри блока
  weekdays: [MON..SAT]      // какие дни недели активны
  periods: Period[]         // пары/слоты в дне: {index, start, end}
}
```

### Дисциплины и их размещение
```
Course {                    // дисциплина
  id, code, title
  prerequisites: CourseId[] // → станут ограничениями порядка блоков
  corequisites: CourseId[]
}

CourseOffering {            // дисциплина, предлагаемая в терме
  id, courseId, termId
  scope: BLOCK | SPAN       // BLOCK = блочная; SPAN = сквозная (через весь терм)  ← гибрид
  allowedBlocks: BlockId[]  // в каких блоках МОЖЕТ идти (для scope=BLOCK)
                            //   пусто = решатель выбирает; один = жёстко закреплена
  maxRoomsPerSection?, ...
}

Section {                   // конкретный поток/группа = то, что назначается
  id, offeringId
  blockId: BlockId          // ★ секция целиком живёт в ОДНОМ блоке (ядро блочности)
  dailyPattern: DailyPattern// какие weekdays + какие periods внутри блока
  capacity: int
  instructorIds: InstructorId[]
  roomIds: RoomId[]         // назначаются микро-решателем
}

DailyPattern {              // «каждый день» = все weekdays блока + нужные пары
  weekdays: [MON..FRI]
  periods: [1,2]            // напр., 1-я и 2-я пара ежедневно
}
```

### Люди и ресурсы
```
Instructor {
  id, name
  blockAvailability: BlockId[]   // в каких блоках доступен
  maxBlocksLoad?: int            // не вести больше N блоков подряд
  unavailable: TimeRange[]       // внутри дневной сетки
}

Room {
  id, capacity, features[]
  availability                   // по блокам и по дневной сетке
}

Student {
  id
  programId                      // учебный план / специальность
  cohortId?                      // когорта, движущаяся вместе
}
```

### Планы и записи
```
StudentRequest {                 // студент должен пройти дисциплину
  id, studentId, courseId
  priority                       // приоритет (как в UniTime sectioning)
}

BlockPlacement {                 // РЕЗУЛЬТАТ решателя для студента
  studentId
  assignments: (blockId → sectionId)[]   // путь студента по блокам
}

Cohort {                         // группа студентов, идущая синхронно
  id, studentIds[]
  curriculumPath: (blockId → courseId)[] // плановая последовательность дисциплин
}
```

**Ключевые инварианты модели:**
1. `Section.blockId` — секция всегда в одном блоке (это и делает блочность нативной).
2. Студент в одном блоке записан на ≤ `maxCoursesPerBlock` секций (для чистого single-subject = 1).
3. `scope = SPAN` позволяет одной модели описывать и блочные, и обычные сквозные курсы → **гибридное расписание** без отдельной подсистемы.

## 3. Двухуровневый решатель

### Layer 1 — Block Assignment (макро)
**Что решает:** в какой блок поставить каждую секцию + по какому пути пустить каждого студента/когорту.
- Переменные: `block(section)`, `block→section` для каждого студента.
- Это задача **секвенирования** на грубой сетке (блоки), а не на минутах.
- Учитывает: пререквизиты (`block(A) < block(B)`), вместимость блоков, нагрузку преподавателей по блокам, правило «одна дисциплина на блок», эл-сть курс↔блок.
- Технологии: CP-SAT / ILP (точно, для разумного размера) или local search (как CPSolver, для масштаба).
- Выход: разбиение «какие секции в каком блоке и кто в них».

### Layer 2 — Intra-Block Timetabling (микро)
**Что решает:** внутри каждого блока — дневное расписание (пары + комнаты) для его секций.
- **Параллельно и независимо по блокам** — главный архитектурный выигрыш.
- Задача маленькая: внутри блока мало секций, студент учит одну дисциплину → конфликтов почти нет.
- По сути классический timetabling (время × комната), но на масштабе одного блока → решается быстро.
- Переиспользуем критерии/ограничения из классической модели (предпочтения комнат, инструкторов, distance и т.д.).

### Связь уровней
Layer 1 ограничивает осуществимость Layer 2 (агрегатная вместимость комнат/преподавателей в блоке). Два способа:
- **Итеративно:** Layer 1 → Layer 2; если блок не разместился — обратная связь (cut) в Layer 1.
- **Агрегатно:** поднять грубые ограничения вместимости (Σ часов секций блока ≤ Σ комната-часов блока) сразу в Layer 1, чтобы Layer 2 почти всегда был осуществим.

## 4. Жёсткие ограничения (block-aware Hard Constraints)

| # | Ограничение | Описание |
|---|---|---|
| H1 | **Одна дисциплина на блок** (на студента) | Студент записан на ≤ `maxCoursesPerBlock` секций в блоке (single-subject = 1). Определяющее правило блочности. |
| H2 | **Секция ⊂ один блок** | `Section.blockId` единственный; dailyPattern помещается в `Block.dayGrid` и даты. |
| H3 | **Порядок по пререквизитам** | `block(prereq).sequence < block(course).sequence`. |
| H4 | **Кореквизиты** | `block(A) == block(B)` (идут в одном блоке). |
| H5 | **Эл-сть курс↔блок** | секция только в `allowedBlocks`. |
| H6 | **Вместимость секции** | записанных ≤ `capacity`. |
| H7 | **Преподаватель: блок-нагрузка** | не ведёт пересекающиеся по дневной сетке секции в одном блоке; уважает `blockAvailability`, `maxBlocksLoad`. |
| H8 | **Комната ≠ 2 секции одновременно** | в пределах дневной сетки блока (микро-уровень). |
| H9 | **Вместимость комнаты ≥ студентов** | как в классике. |
| H10 | **Доступность ресурсов** | преподаватель/комната доступны в блоке и в нужных парах. |
| H11 | **Сквозные курсы (SPAN)** | секция scope=SPAN занимает ресурс во всех блоках, которые перекрывает (мост к классике). |

## 5. Мягкие критерии (block-aware Optimization Objectives)

| Критерий | Что измеряет |
|---|---|
| **Удовлетворённость запросов по приоритету** | максимум пройденных дисциплин, высокий приоритет доминирует (как UniTime sectioning). |
| **Соответствие учебному плану (curriculum flow)** | штраф за отклонение от педагогически желаемого порядка дисциплин по блокам. |
| **Равномерность нагрузки преподавателя по блокам** | избегать «4 блока подряд, потом простой»; сглаживание. |
| **Целостность когорты** | держать когорту вместе на одном пути по блокам. |
| **Без «дыр» в плане студента** | минимизировать пустые блоки (если нежелательны). |
| **Баланс заполнения параллельных секций** в блоке | равномерно по эквивалентным секциям. |
| **Спейсинг связанных дисциплин** | напр., теория и её практика — в смежных блоках (back-to-back) или с нужным зазором. |
| **Предпочтения время/комната** внутри блока | переиспользуем микро-критерии классики (too-big rooms, distance, useless slots…). |
| **Minimal Perturbation (MPP)** | при переплане — минимум изменений к уже опубликованному. |

Иерархия (предлагаемая): пререквизиты/одна-на-блок (hard) ≫ удовлетворённость запросов ≫ curriculum flow ≫ нагрузка преподавателей / целостность когорты ≫ микро-предпочтения (комнаты/время) ≫ баланс/спейсинг.

## 6. Чем это лучше «UniTime-подхода»

| Аспект | UniTime (обходное) | UniPlan (нативное) |
|---|---|---|
| Блок | `time pattern ∩ date pattern`, артефакт | сущность 1-го класса с порядком и дневной сеткой |
| «Одна дисциплина на блок» | моделируется конфликтами вручную | жёсткое ограничение H1 |
| Последовательность дисциплин | distribution `Following Weeks`/`Precedence` поверх дат | `Block.sequence`, нативный порядок |
| Пререквизиты как порядок блоков | нет прямой связи | H3: `block(A) < block(B)` |
| Масштабируемость | один большой семестровый солвер | декомпозиция → блоки параллельно |
| Гибрид (блок + сквозные) | разные date pattern, костыли | `scope: BLOCK | SPAN` в одной модели |

## 7. Открытые решения (нужно зафиксировать)

1. **Стек и технология решателя.** Layer 1: CP-SAT (OR-Tools) / ILP — точно, до ~среднего размера; либо local search (по образцу CPSolver) — для большого масштаба. Layer 2 параллелизуется в любом случае. → влияет на выбор языка (Python+OR-Tools vs JVM+своё).
2. **`maxCoursesPerBlock`** — строго 1 (чистый single-subject) или допускаем 2 (block + поддерживающий курс)?
3. **Экзамены/аттестация по блокам** — экзамен в конце каждого блока как отдельная сущность (адаптация exam-модели UniTime под блок) или часть последних дней блока?
4. **Переменная длина блоков** — поддерживаем разную `lengthDays`, или фиксируем равные?
5. **Когорты vs индивидуальные пути** — основной режим: студенты движутся когортами (проще) или каждый сам по себе (гибче, тяжелее)?

## 7.1. Будущие возможности и как их учитывает архитектура

Три запрошенные возможности. Все **допустимы** в Strategy A; важно отделить «движок даёт нативно» от «строим в прикладном слое».

### (1) Импорт старых PDF-расписаний ради «минимальных изменений»
Распадается на две независимые задачи:
- **Минимальность изменений = MPP.** Это нативная способность CPSolver: загрузить старое расписание как *initial solution* и минимизировать `Perturbations`. **Но MPP работает от СТРУКТУРИРОВАННЫХ данных, а не от PDF.** Поскольку мы и так версионируем расписания (R14), «минимальное изменение» работает само собой, когда UniPlan — система записи; PDF для этого не нужен.
- **Парсинг PDF — отдельный, хрупкий ingestion-компонент**, не имеющий отношения к решателю. Те таблицы (объединённые ячейки, наборы недель в скобках, две подгруппы в одной ячейке) парсятся ненадёжно. Инструменты на JVM: PDFBox (текст), Tabula (таблицы), Tesseract (если сканы). 
- **Рекомендация:** не связывать MPP с PDF. PDF-импорт — это **разовый ассистируемый bootstrap легаси** (извлечь → человек проверяет/правит → в структурированную модель), а не основной поток. Дальше «минимальные изменения» обеспечивает версионирование + MPP без PDF.
- Статус: MPP — ✅ нативно (пост-MVP); PDF-импорт — ⚠️ отдельный assisted-import, не в ядре.

### (2) Самообслуживание преподавателей (слоты «удобно/неудобно» + одобрение составителем)
- **Сторона решателя — ✅ нативно.** Недоступные слоты = жёсткое ограничение (убираются из домена); «удобно/неудобно» = мягкие **time preferences** по шкале −4…+4/R/P (ровно модель UniTime, см. `UniTime-ограничения-и-критерии.md`). «Программа строит ответное расписание» = обычный прогон, потребляющий одобренные предпочтения.
- **Сторона приложения — это реальная фича продукта** (не бесплатно): роль `Instructor`, UI ввода доступности, **очередь одобрения** у составителя, статус предпочтения (`proposed/approved/rejected`), хранение. Архитектура SaaS (Spring Boot, роли) это **вмещает**, но строить надо. Концептуально хоженая тропа: в UniTime кафедры/преподаватели тоже вводят предпочтения, потом центральное составление.
- Статус: ⚠️ архитектура вмещает; это фаза 2+, не в 5-недельном MVP.

### (3) Много институтов в одном вузе (разграничение)
- **Не путать с мульти-тенантностью.** Тенант = **вуз**; институты — это **оргиерархия ВНУТРИ тенанта**: `University → Institute/Факультет → Department/Кафедра → Group`. (В документах уже видно «каф. Факультетской терапии **ИКМ**» = Институт клинической медицины.)
- **Разграничение = scoping + авторизация:** составитель видит/правит данные своего института. Это в точности модель UniTime — «распределённая система, где много кафедральных менеджеров координируют общий план» (solver groups, committed solutions).
- **Общие ресурсы между институтами** (аудитории, преподаватели, **клинические базы**) — ключевой нюанс: расписания нельзя силосить полностью, есть слой общих ресурсов. Решение (как в UniTime): либо общий прогон, либо силосные прогоны с чужими расписаниями как *committed* (зафиксированными) ограничениями. Для нашего клинического случая базы (кафедры) шарятся между институтами → нужна координация общих ресурсов.
- Статус: ✅ архитектура вмещает (оргиерархия + scoping + общие ресурсы); силос-vs-общий прогон — отдельное решение, пост-MVP.

### Дешёвая страховка — заложить в схему MVP уже сейчас (чтобы не ретрофитить)
Сами фичи — пост-MVP, но эти поля в модель добавить **сейчас** (дёшево добавить, дорого ретрофитить):
- **Версионирование расписаний** (уже планируется) → даёт «минимальные изменения» без PDF.
- **Оргиерархия**: FK `University(tenant) → Institute → Department` на сущностях, даже если MVP работает с одним институтом.
- **Владение/шаринг ресурсов**: у `Location/Instructor/Base` — владеющий орг-юнит + флаг «общий», чтобы межинститутный шаринг был выразим позже.
- **Преподавательские предпочтения как данные** + статус `proposed/approved` (заготовка под воркфлоу).
