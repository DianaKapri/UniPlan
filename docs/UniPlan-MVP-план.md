# UniPlan — план MVP (5 недель, Strategy A)

Решение по архитектуре: [`ADR-0001-форк-cpsolver.md`](./ADR-0001-форк-cpsolver.md) — **depend + extend CPSolver на JVM**.
Доменная модель: [`UniPlan-блочное-расписание-дизайн.md`](./UniPlan-блочное-расписание-дизайн.md).

## Ключевое упрощение MVP: где solver, а где данные

Самое важное решение по объёму. В блочных программах последовательность «какой курс в каком блоке» обычно **задана учебным планом** (curriculum), а не вычисляется. Поэтому в MVP:

- **Layer 1 (назначение блоков) = ДАННЫЕ**, а не оптимизация. Когорта несёт готовый план `(блок → курс)`. Solver Layer 1 (выбор блока, когда есть свобода) — **после MVP**.
- **Layer 2 (расписание дня внутри блока) = SOLVER.** Здесь переиспользуем `coursett` из CPSolver: внутри каждого блока расставляем пары + комнаты + преподавателей. Это та часть, где движок реально нужен, и она уже написана и проверена.

Это снимает необходимость писать собственную IFS-модель к MVP и делает срок реалистичным.

## Рамки MVP (жёстко режем)

✅ В MVP: когортный режим (студенты идут группами), curriculum-driven назначение блоков, одна конфигурация курса (без lecture/lab subparts), per-block solve через coursett, две оси визуализации, базовый SaaS (auth + один тенант).

❌ Не в MVP: индивидуальные пути студентов, Layer-1 оптимизация, экзамены, student sectioning, MPP/переплан, сложная мультитенантность, интеграции (LMS/SIS).

## Стек

| Слой | Выбор | Почему |
|---|---|---|
| Язык | **Kotlin** (JVM) | приятнее Java, 100% интероп с CPSolver |
| Backend | **Spring Boot** (REST) | зрелый, годен и для SaaS, и для on-prem-пакета |
| Solver | **CPSolver** как Maven-dep `org.unitime:cpsolver`, прогон в async-джобе | долгие прогоны не должны блокировать API |
| БД | **PostgreSQL** | мультитенант-дружелюбна |
| Frontend | минимальный: **React + grid/календарь** (или server-rendered, если жмёт время) | две оси: цепочка блоков + дневная сетка |
| Деплой | один артефакт (jar/контейнер); SaaS-инстанс сейчас, тот же артефакт on-prem позже | |

> On-prem-готовность с самого начала: блочный код — в наших пакетах; CPSolver — внешняя зависимость; внутренности не патчим → LGPL чисто.

## Самый рискованный кусок → de-risk в неделю 1

**Адаптер «домен UniPlan → модель coursett»** (блок как контекст, дневная сетка как time, комнаты/преподаватели). Сделать спайк-прогон coursett на игрушечном входе в первые дни — это снимает главный технический риск до того, как на нём завязан весь продукт.

## План по неделям

### Неделя 1 — фундамент + спайк solver-а
- Репозиторий, Kotlin/Gradle, Spring Boot скелет, Postgres, CI.
- Доменные сущности: `Term, Block(sequence, dayGrid), Course, CourseOffering, Section(blockId, dailyPattern), Instructor, Room, Cohort` + миграции.
- Подключить `org.unitime:cpsolver`; **спайк**: прогнать `coursett` на игрушечном входе одного блока, убедиться что получаем назначения время+комната.

### Неделя 2 — данные и ввод
- CRUD API для сущностей; базовый ввод (комнаты, преподаватели, курсы, когорты).
- Календарь блоков: `Term → Block[]` с `sequence` и дневной сеткой.
- Curriculum когорты: упорядоченный план `(блок → курс)` (это и есть Layer 1 как данные) + валидация (пререквизиты `block(A)<block(B)`, одна дисциплина на блок).

### Неделя 3 — интеграция solver-а (Layer 2 по блокам)
- Адаптер домен → coursett: секции блока → классы с дневными time-локациями, комнаты, преподаватели.
- Прогон per-block как async-джоба; сохранение результата (секция → пара+день+комната).
- Жёсткие ограничения H6–H10 через coursett; H1/H3 — валидацией данных.
- Параллельный прогон блоков (они независимы).

### Неделя 4 — UI и вывод
- Визуализация двух осей: цепочка блоков + дневная сетка выбранного блока.
- Виды: когорта (путь по блокам), преподаватель, комната.
- Кнопка «составить», прогресс, показ конфликтов/неназначенных.

### Неделя 5 — SaaS-обвязка + буфер (+1 неделя)
- Auth, один тенант, деплой SaaS.
- Ручные правки/переопределения, повторный прогон.
- Буфер на проскальзывание (закладываем заранее — срок агрессивный).

## После MVP (бэклог)
Layer-1 оптимизация (выбор блока) как IFS-модель · индивидуальные пути студентов · student sectioning · экзамены по блокам · MPP/переплан · мультитенантность · интеграции.
