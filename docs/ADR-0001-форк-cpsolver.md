# ADR-0001: Строить ли UniPlan вокруг CPSolver (форк) вместо solver-а с нуля

Статус: **ПРИНЯТО → Strategy A (depend + extend)** · Дата: 2026-06-21

## Решение (зафиксировано)

Выбран **вариант A: depend + extend CPSolver на JVM**. Обоснование по входным данным:
- **JVM приемлем** → A технически возможен.
- **SaaS сейчас, on-prem в будущем** → при depend+extend (без буквального форка) блочная модель лежит в своих пакетах и остаётся проприетарной; для SaaS обязательств LGPL почти нет, для будущего on-prem достаточно поставлять LGPL-jar и дать пере-слинковать. **Правило: внутренности CPSolver не патчим**; если придётся — тонкий отдельный модуль.
- **MVP за 5 недель** — решающее: свой конкурентный solver за этот срок нереалистичен; переиспользуем `coursett` для Layer 2 и не пишем движок с нуля.

См. план реализации: [`UniPlan-MVP-план.md`](./UniPlan-MVP-план.md).

## Контекст

Альтернатива «писать всё с нуля»: взять **CPSolver** (движок, на котором работает UniTime) и построить продукт вокруг него.

Проверенные факты:
- **CPSolver — LGPL-3.0**, 100% Java, Maven/Ant.
- Структура: **IFS** (Iterative Forward Search) — общий движок локального поиска (variables / values / constraints / criteria, conflict-based statistics, hill climbing / great deluge / simulated annealing). Поверх него — доменные модели `coursett`, `exam`, `studentsct`, `instructor`.
- Фреймворк IFS рассчитан на **расширение**: можно объявить свои `Variable`/`Value`/`Constraint`/`Criterion` и собрать новую доменную модель, не модифицируя существующие.

## Главный реframe

**UniTime — это уже «продукт, построенный вокруг CPSolver».** То есть выбранный путь = повторить то, что сделал UniTime (слой приложения вокруг движка), но со своим приложением и своей доменной моделью (блочность).

И второе: **«форкнуть» ≠ «использовать».** Есть три разных уровня связи, и их часто путают:
1. **Depend (зависимость)** — подключить как Maven-артефакт `org.unitime:cpsolver`, написать свою модель на IFS. Апстрим-обновления прилетают, твой код остаётся твоим.
2. **Extend (расширить)** — то же + свои классы модели в своих пакетах. Это «правильный форк по духу».
3. **Fork (буквальный форк)** — скопировать репозиторий и расходиться. Теряешь апстрим, берёшь на себя поддержку всего движка. Нужен **только** если правишь внутренности движка.

> Вывод по умолчанию: **depend + extend**, а не буквальный fork. Форкать только то, что вынужден патчить.

## Что лицензия LGPL-3.0 реально разрешает

- **Можно** использовать в проприетарном/коммерческом продукте как линкуемую библиотеку. Твой код приложения и твоя доменная модель **не заражаются** copyleft.
- **Но** изменения в самих файлах CPSolver — copyleft: их надо открыть под LGPL. (Поэтому depend+extend выгоднее форка — пишешь рядом, а не внутри.)
- **SaaS:** запуск на своём сервере (пользователи ходят по сети) по LGPLv3 **не считается distribution** → никаких обязательств раскрытия. (Было бы иначе при AGPL — но тут LGPL.)
- **On-premise/desktop поставка:** обязан (а) дать исходники LGPL-части + свои правки к ней, (б) позволить пользователю пере-слинковать с изменённой версией библиотеки. Проприетарный код приложения при этом остаётся закрытым.

→ Юридически путь чистый для коммерческого продукта.

## Что ты переиспользуешь vs что всё равно пишешь сам

| Переиспользуешь (большой выигрыш) | Всё равно строишь сам (≈80% продукта) |
|---|---|
| Движок локального поиска IFS (годы R&D) | Ввод/управление данными (курсы, комнаты, преподаватели) |
| Conflict-based statistics, метаэвристики | UI / UX, визуализация расписания |
| Готовый каталог ограничений и критериев (`coursett`/`studentsct`/`exam`) | Workflow, роли, права, публикация |
| Микро-таймтейблинг внутри блока почти «из коробки» | Импорт/экспорт, интеграции (LMS, SIS) |

**Главное отрезвление:** solver — это самая трудная и рискованная часть (написать конкурентный таймтейблер с нуля — это многолетний R&D), но это лишь ~20% продукта. Форк экономит движок, а не продукт.

## Блок-специфичный нюанс (важно для UniPlan!)

Сила CPSolver — масштабировать локальный поиск на **гигантские недельные расписания всего вуза**. Но наша архитектура (см. [`UniPlan-блочное-расписание-дизайн.md`](./UniPlan-блочное-расписание-дизайн.md)) **декомпозирует** задачу:
- Layer 2 (внутри блока) — маленький, конфликтов мало.
- Layer 1 (назначение блоков + порядок) — комбинаторная задача на грубой сетке.

→ После декомпозиции каждая под-задача **небольшая**. А значит главное преимущество CPSolver (масштаб) нужно меньше — это **ослабляет** аргумент «наследовать движок» и **усиливает** вариант «писать чисто на CP-SAT (OR-Tools)», который для Layer 1 (секвенирование, пререквизиты, оптимум) выражается декларативно чище и решается точно.

То есть для **блок-first** продукта развилка реальная, а не очевидная.

## Три стратегии

### A. Depend + extend IFS (если JVM ок)
Подключить CPSolver, переиспользовать `coursett` для Layer 2, написать свою блочную модель для Layer 1 на IFS.
- ➕ Быстрее всего до правдоподобного движка; готовый каталог критериев; зрелый код.
- ➖ Лок-ин на JVM (Java/Kotlin/Scala); research-grade код со скудной докой; крутая кривая входа; ops JVM.

### B. Solver-as-a-service
CPSolver (extended) как микросервис за API; приложение — на любимом стеке (Python/TS).
- ➕ Развязывает стек приложения и движок; движок переиспользован.
- ➖ Больше движущихся частей и эксплуатации; границы API надо проектировать.

### C. Reference only → строить на OR-Tools / CP-SAT
Код CPSolver не берём, берём **знание** (наши доки по ограничениям/критериям) и строим модель на CP-SAT.
- ➕ Свобода стека (Python), современный тулинг, чистая блочная модель, точное решение Layer 1; CP-SAT отлично поддерживается.
- ➖ Переписываем поиск/критерии; меньше «боевого пробега», но CP-SAT сам по себе зрелый и мощный.

## Рекомендация

- **Если важна скорость до рабочего движка и команда ок с JVM →** вариант **A** (depend+extend, не буквальный fork). Берёшь движок и каталог критериев, добавляешь блочный домен.
- **Если приоритет — современный стек, чистая блок-first модель и точность Layer 1 →** вариант **C** на CP-SAT, используя CPSolver и наши доки как референс. Декомпозиция по блокам делает этот путь реалистичным (масштаб CPSolver не критичен).
- **B** — если уже знаешь, что приложение будет не на JVM, но хочешь движок CPSolver.

Чего точно **не** делать: буквально форкать и расходиться с апстримом без явной нужды патчить внутренности.

## Что решает развилку (открытые вопросы)
1. **Стек команды** — JVM приемлем? (определяет A/B vs C)
2. **SaaS или on-premise** — влияет на обязательства LGPL (для SaaS — почти нулевые).
3. **Open-source или проприетарный** продукт?
4. **Масштаб** — вуз в десятки тысяч записей (аргумент за зрелый движок) или меньше (аргумент за чистый CP-SAT)?
5. **Срок до MVP** — жёсткий (→ A, переиспользовать максимум) или есть время на свою модель (→ C)?
